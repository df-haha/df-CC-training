3.1 三種操作模式深度講解

Claude Code 的三種模式

想像你請了一個工人來家裡裝修：

模式 比喻 說明
Normal 每做一步都問你 「我要拆這面牆，可以嗎？」
Auto-accept 放手讓他做 直接拆牆，不問你
Plan 先畫設計圖 「我打算這樣改，你看看？」

Normal 模式特點

特點

每次修改檔案都會問你

每次執行指令都會問你

最安全，適合學習階段

適合情況：

  • 剛開始用 Claude Code
  • 重要專案，不能出錯
  • 想了解 Claude 在做什麼

Auto-accept 模式

特點

速度快，流暢

可能改錯、刪錯

Auto-accept 風險警示

Auto-accept 模式下，Claude 可能會：改錯檔案、執行危險指令、刪除重要資料

建議搭配：

  • Git 版本控制（可以 rollback）
  • Permission 設定（禁止危險操作）

適合情況：

  • 熟悉 Claude Code 後
  • 簡單重複的任務
  • 有 Git 版本控制保底

3.2 Plan Mode 深度操作

Plan Mode 完整工作流程

步驟 操作
1. 進入 Plan Mode Shift+Tab 切換，或輸入 /plan
2. 描述需求 「我想加一個會員系統」
3. Claude 提出計畫 列出步驟、需要修改的檔案、技術選型
4. 討論修正 「第 2 點改用 Context 而不是 Redux」
5. 確認計畫 「OK，就這個計畫」
6. 切回 Normal 模式 Shift+Tab 切換
7. 執行 「開始執行計畫」

何時使用 Plan Mode？

適合用 Plan Mode

新功能開發涉及多個檔案，需要先規劃
重構程式碼需要考慮影響範圍
架構變更需要評估方案
複雜 Bug 修復需要先分析原因
不確定怎麼做讓 Claude 先提案
VS

不需要 Plan Mode

改錯字直接改就好
加 console.log一句話搞定
簡單樣式調整直接做
已經很明確的任務不用再規劃

Plan Mode 小結

Plan Mode 核心價值

1. 先思考再動手 - 避免盲目開工

2. 可以討論修改 - 計畫不滿意可以調整

3. 掌握全局 - 知道會改哪些檔案、影響範圍

4. 減少錯誤 - 發現問題在計畫階段而非執行後

3.3 Permission 設定完整指南

真實災難案例：AI Agent 誤刪整個 C 槽

某開發者讓 AI Agent 自動執行清理任務，Agent 誤判指令，執行了類似 rm -rf C:\ 的命令。結果：整個系統毀損，資料全部遺失！

問題出在哪？

  • 使用 Auto-accept 模式（AI 做什麼都自動同意）
  • 沒有設定 Deny 規則（沒有「禁止清單」）
  • 沒有任何安全網

教訓：權限設定不是選配！

核心觀念

Claude Code 能直接操作你的電腦

包括：修改檔案、執行指令、刪除資料

權限設定是保護你電腦的最後一道防線

即使你用 Normal 模式，設定 Deny 規則仍然很重要！
因為你可能不小心點了「接受」

三層權限控制

層級 說明 安全性
層級 1 互動式確認（Normal 模式） 最安全，每次都問你
層級 2 Permission 設定（白名單 + 黑名單） 可控，允許/禁止特定操作
層級 3 Bypass（跳過所有檢查） 超危險！不建議使用

層級 1：Normal 模式（預設）

Claude
我要執行 npm test，可以嗎？
你
可以
Claude
我要刪除 temp.txt，可以嗎？
你
可以
Claude
我要執行 rm -rf /，可以嗎？
你
不可以！！！

優點：最安全 | 缺點：每次都要點確認

層級 2：Permission 設定

用 /permissions 指令設定白名單和黑名單：

/permissions

Allow（白名單）- 不用問我

npm test

npm run build

npm run dev

git status

git add

git commit

VS

Deny（黑名單）- 絕對禁止

rm -rf

rmdir /s /q C:\

npm publish

git push --force

drop database

format c:

Permission 設定效果

Claude
我要執行 npm test
[自動執行，不問你，因為在白名單]
Claude
我要執行 rm -rf node_modules
這個指令被禁止了
[不會執行]

層級 3：Bypass（危險！）

# 跳過所有安全檢查
claude --dangerously-skip-permissions
這個指令會：跳過所有權限檢查、不問你任何確認、Claude 想做什麼就做什麼。強烈不建議使用！

唯一合理的使用情況：

  • 在完全隔離的測試環境
  • 你知道自己在做什麼
  • 做好完整備份

Permission 最佳實踐（1/2）

1. 設定合理的白名單

類型 建議加入白名單的指令
npm/yarn npm test、npm run dev、npm run build
Git 查看 git status、git diff、git log
檔案查看 dir、type、more

Permission 最佳實踐（2/2）

2. 設定重要的黑名單

類型 一定要禁止的指令
刪除操作 rm -rf、rmdir /s /q、系統槽操作
發布操作 npm publish、docker push
Git 危險操作 git push --force、git reset --hard
資料庫操作 drop database、truncate table
即使設定了 Permission，也建議搭配 Git 版本控制，出錯可以 rollback

實作練習

練習 1：查看目前設定

  • 執行 /permissions 查看目前的權限設定

練習 2：新增 Allow 規則

  • 新增一個 Allow 規則：npm test

練習 3：新增 Deny 規則

  • 新增一個 Deny 規則：rm -rf

練習 4：測試效果

  • 請 Claude 執行 npm test（應該自動執行）
  • 請 Claude 執行 rm -rf temp（應該被拒絕）

本單元重點回顧

核心觀念

1. 三種模式 - Normal（最安全）、Auto-accept（要小心）、Plan（先規劃再執行）

2. Plan Mode 使用時機 - 新功能、重構、架構變更、不確定怎麼做時

3. Permission 設定 - 一定要設黑名單（禁止危險指令）

4. 安全原則 - Claude 能直接操作電腦，權限設定不是選配！

安全原則總結

原則 做法
新手階段 保持 Normal 模式，看清楚 Claude 在做什麼
重要專案 Normal 模式 + 設定 Deny 規則
複雜任務 用 Plan Mode 先規劃
日常開發 搭配 Git 版本控制
永遠記住 Claude 能真的執行，不只是給建議！
