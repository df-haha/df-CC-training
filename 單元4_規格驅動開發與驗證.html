4.1 SDD 核心概念

什麼是 SDD？

SDD = Spec-Driven Development = 規格驅動開發

先把需求講清楚，再讓 AI 幫你寫程式

用白話說：

  • 不要一開口就叫 Claude 寫程式
  • 先把「要什麼」講清楚
  • 確認雙方理解一致，再開始動手

傳統做法 vs SDD 做法

傳統做法

你：幫我寫一個登入功能

Claude：（直接開始寫...）

你：不對，我要的不是這樣...

Claude：（重寫...）

你：還是不對...

來回修改，浪費時間

SDD 做法

你：我要做登入功能，先別寫，幫我釐清需求

Claude：好的，我有幾個問題...

你：（一一回答）

Claude：了解，我整理一下規格...

你：對，就是這樣

一次到位！

SDD vs TDD vs BDD

開發方法 核心概念 流程 適合情境
TDD 測試先行 先寫測試 → 再寫程式 品質要求高
BDD 行為描述 用「使用者故事」描述功能 需求複雜
SDD 規格描述 用自然語言描述需求 AI 協作

為什麼 SDD 最適合 Claude Code？

TDD / BDD 的問題

TDD：要先會寫測試，Claude 不知道你要測什麼

BDD：需要特定格式（Given-When-Then），學習成本高

VS

SDD 的優勢

用自然語言就好

Claude 會幫你問清楚

不需要額外格式

SDD 三原則（一）：具體描述

不好的描述

「幫我寫一個表單」

好的描述

「幫我寫一個註冊表單，包含：

  • 姓名（必填，2-20 字）
  • Email（必填，要驗證格式）
  • 密碼（必填，8 位以上，要有大小寫和數字）
  • 確認密碼（要和密碼一致）
  • 送出後顯示成功訊息」

SDD 三原則（二）：分步拆解

不好的做法

「幫我做一個完整的電商網站」

太大，Claude 不知道從何下手

好的做法

「我要做電商網站，我們分步來：

  • 第一步：先做商品列表頁
  • 第二步：做商品詳情頁
  • 第三步：做購物車

一步一步，確保每步都對

SDD 三原則（三）：先探索再執行

不好的做法

直接說「幫我改這個 Bug」

好的做法

「這裡有個 Bug，先別動，幫我分析一下：

  1. 現在的程式碼在做什麼？
  2. Bug 的原因可能是什麼？
  3. 你打算怎麼修？」

先理解現況，再動手

核心技巧：讓 Claude 當需求顧問

魔法咒語

「先不要寫，幫我釐清需求」

這句話會讓 Claude：

  • 停下來思考
  • 列出需要確認的問題
  • 等你回答後才開始動手

SDD 核心概念小結

記住三件事

1. 具體描述：說清楚要什麼，不要讓 Claude 猜

2. 分步拆解：大任務切小，一步一步來

3. 先探索再執行：先理解再動手，避免做白工

4.3 驗證是最重要的事

Boris 的名言

「給 Claude 驗證方式，品質提升 2-3 倍。
這是最重要的事。」
— Boris (Claude Code 專家)

沒有驗證的問題

你
幫我寫一個 formatPrice 函式，把數字變成價格格式
Claude
好的！

function formatPrice(num) { return '$' + num.toFixed(2); }

完成了！
上線後用戶反映：「為什麼 1000 元顯示成 $1000.00 而不是 $1,000.00？」
沒有千分位！如果有驗證，馬上就會發現。

有驗證的效果

你
幫我寫一個 formatPrice 函式

驗證測試：
- formatPrice(1234) → "$1,234"
- formatPrice(1000000) → "$1,000,000"
- formatPrice(99.5) → "$100"（四捨五入）
- formatPrice(0) → "$0"

寫完後跑這些測試，全部通過才算完成

有驗證的效果（結果）

Claude
好的！讓我寫並測試...

function formatPrice(num) { return '$' + Math.round(num).toLocaleString(); }

測試結果：
✓ formatPrice(1234) → "$1,234"
✓ formatPrice(1000000) → "$1,000,000"
✓ formatPrice(99.5) → "$100"
✓ formatPrice(0) → "$0"

全部通過！
有驗證，Claude 自己會檢查；不對會自己修正；品質保證！

核心概念

Claude 需要「標準答案」來自我檢查

沒有驗證 = Claude 不知道做對沒 → 可能產出「看起來對但其實錯」的程式碼

有驗證 = Claude 可以自己確認 → 不對會自己修正 → 品質大幅提升

不同情境的驗證方式

任務類型 驗證方式 具體範例
寫程式 跑測試 npm test、寫測試案例
改 UI 截圖 改完截圖讓你確認
寫 API curl 測試 curl localhost:3000/api/xxx
修 Bug 重現→修→驗證 確認錯誤消失
效能優化 Benchmark 比較前後數據

驗證測試：
- validateEmail("test@example.com") → true
- validateEmail("invalid") → false
- validateEmail("test@.com") → false
- validateEmail("") → false

測試全過才算完成

4.4 Superpowers 思維

AI 的問題：太急著動手

你有沒有遇過這種同事？

你
幫我做一份報告
同事
（聽到一半就開始做）
做好了！
你
...這不是我要的
AI 也有這個問題！很聰明，但缺乏紀律——太急著動手，沒有先搞清楚需求。

Superpowers 是什麼？

簡單說：給 AI 一本「員工手冊」

就像公司會給新人一本手冊，告訴他：

  • 接到任務先做什麼
  • 怎麼確認需求
  • 怎麼回報進度
  • 什麼事情不能做

Superpowers 就是這樣的手冊，讓 AI 變得更有紀律。

Superpowers 概念 1：先問清楚再動手

沒有 Superpowers 的 AI

你：幫我做一個表單

AI：好！（馬上開始寫）...做好了！

你：我要的不是這種表單...

VS

有 Superpowers 的 AI

你：幫我做一個表單

AI：好的，在開始之前，讓我確認：
1. 這個表單是要收集什麼資料？
2. 需要哪些欄位？

確認清楚才動手

Superpowers 概念 2：先寫驗收標準

比喻：裝修房子

不好的做法

師傅：「交給我！」（開始裝修）

三天後...

師傅：「好了！」

你：「這顏色不對、尺寸不對...」

好的做法

師傅：「開始前，我們先確認驗收標準」

  • 牆壁顏色：白色 ✓
  • 櫃子尺寸：寬 120cm ✓
  • 完成後我會拍照確認 ✓

邊做邊對照標準

用在 AI Coding

你
幫我寫一個計算折扣的功能

驗收標準：
- 原價 1000，打 8 折 → 800
- 原價 500，打 9 折 → 450
- 原價 0，打任何折 → 0
- 折扣超過 100% → 顯示錯誤
Claude
了解！我會：
1. 先寫這些測試
2. 然後寫程式
3. 確認所有測試都通過

這樣你就知道我做對了！

Superpowers 概念 3：不說「應該」

一般 AI 的說法

「這樣應該可以」

「大概是這樣」

「理論上沒問題」

VS

有 Superpowers 的 AI

「我已經測試過，結果如下：

✓ 測試 1 通過

✓ 測試 2 通過

全部通過，確認完成！」

Superpowers 概念 4：堵住藉口

AI 很聰明，會找各種理由偷懶。

AI 的藉口 反駁
「這個太簡單，不用測試」 越簡單越應該測試，證明給我看
「測試會拖慢速度」 不測試出錯重做更慢
「這個改動很小」 小改動也可能有大影響
「我很確定沒問題」 確定就跑測試證明，不確定更要測
在 Superpowers Skills 裡，這些藉口都被預先堵住了

實際應用：在 CLAUDE.md 加入規則

## 工作規則

1. 接到任務先釐清需求，一次問一個問題
2. 開始做之前，先確認「怎麼驗收」
3. 做完要實際測試，不能說「應該可以」
4. 有任何不確定的地方，先問我

這樣每次對話，Claude 都會遵守這些紀律。

Superpowers 思維總結

四個核心概念

1. 先問清楚，別急著做 → 一次問一個問題，用選擇題

2. 先講好怎麼驗收，再開始做 → 做完要能證明「做對了」

3. 不說「應該」，要說「我測試過了」 → 有證據才算數

4. 堵住偷懶的藉口 → 不給 AI 繞過規則的機會

4.5 SDD 完整流程與回顧

SDD 五步驟流程

步驟 做什麼 關鍵動作
Step 1 描述需求 用自然語言說「我想做 XXX」
Step 2 讓 Claude 反問 「先不要寫，幫我釐清需求」
Step 3 確認規格 Claude 整理成規格文件，你確認
Step 4 分步執行 一步一步做，每步驗收
Step 5 驗證結果 執行測試、手動確認

驗證範本整理

任務類型 範本
寫程式 「驗證測試：輸入 A → 預期輸出 X」
改 UI 「改完截圖讓我確認」
修 Bug 「先重現 Bug，修完後驗證問題消失」
寫 API 「用 curl 測試，確認回傳正確」
不知道怎麼驗證 「這個功能應該怎麼驗證？」

實用咒語整理

情境 咒語
開始新功能 「我想做 XXX，先不要寫，幫我釐清需求」
不確定怎麼做 「有幾種做法？各有什麼優缺點？」
要修 Bug 「先分析這個 Bug 的原因，再告訴我你打算怎麼修」
要重構 「先看一下現在的程式碼，告訴我有什麼問題」
確認理解 「你理解的需求是什麼？先說一遍讓我確認」
