2.1 Prompt Engineering vs Context Engineering

為什麼要先講這個？

在學習 CLAUDE.md 和 Context 管理之前，
我們要先搞懂一個更大的概念：

核心觀念

你和 AI 互動的方式，正在從「寫好問題」進化到「管理記憶」

Prompt Engineering（提示工程）

定義：設計單次輸入，讓 AI 產出最佳回應

技巧 說明 範例
角色設定 告訴 AI 扮演什麼角色 「你是一個資深 React 工程師」
任務描述 清楚說明要做什麼 「幫我寫一個登入表單元件」
輸出格式 指定回答格式 「用 markdown 表格呈現」
Few-shot 給幾個範例 「範例：輸入 apple → 蘋果」

Prompt Engineering 的問題

你
我要做一個 React 專案，用 TypeScript，用 Tailwind...
幫我寫一個 Button 元件
5 分鐘後，新對話
我要做一個 React 專案，用 TypeScript，用 Tailwind... ← 又要說一次！
幫我寫一個 Input 元件
10 分鐘後，新對話
我要做一個 React 專案... ← 每次都要重複！

這就像每天上班都要重新自我介紹一樣累

Context Engineering（脈絡工程）

定義：管理 AI 的「工作記憶」，讓它持續理解你的專案

CLAUDE.md 已經寫好
- 這是 React 專案
- 用 TypeScript
- 用 Tailwind CSS
- 元件放 src/components/
你只要說
「幫我寫一個 Button 元件」

Claude 自動知道要用 React + TypeScript + Tailwind，
而且會把檔案放在正確的位置。

兩者的核心差異

維度 Prompt Engineering Context Engineering
關注什麼 「這句話怎麼說」 「AI 腦中有什麼」
時間範圍 單次對話 整個專案週期
目標 得到精準回答 持續高品質協作
工具 提示詞技巧 CLAUDE.md、/compact、/clear
比喻 寫一封好的 Email 建立長期合作關係

為什麼 Context Engineering 越來越重要？

1. AI Agent 時代來臨

以前

你問一個問題，AI 回答一次，結束

→

現在

AI Agent 持續幫你：寫程式、跑測試、修 Bug、部署

需要「長期記憶」才能連貫工作

為什麼 Context Engineering 越來越重要？

2. Context 決定品質上限

Context A

只有這段程式碼

→ AI 給出通用建議

VS

Context B

有整個專案背景、效能要求、現有架構

→ AI 給出針對性的最佳方案

同樣的 Prompt，不同 Context，結果天差地遠！

為什麼 Context Engineering 越來越重要？

3. 複利效應

好的 Context 管理 = 知識累積
第 1 天：設定好 CLAUDE.md
第 2 天：Claude 已經知道專案背景
第 3 天：Claude 記得之前的設計決策
...
第 30 天：Claude 對你的專案瞭若指掌

越用越順，這就是複利

白話總結

核心差異

Prompt Engineering = 「怎麼問問題」

→ 單次互動的技巧，像面試時怎麼回答問題

Context Engineering = 「AI 腦中有什麼」

→ 長期協作的基礎，像和同事建立默契

兩者都重要，但 Context Engineering 是未來趨勢！

2.2 CLAUDE.md 入門

什麼是 CLAUDE.md？

CLAUDE.md 是放在專案根目錄的設定檔，
Claude 每次開始對話時都會自動讀取。

作用

讓 Claude 一開始就知道這個專案的背景

就像新同事第一天上班，你給他一份「專案說明文件」，
他馬上就能進入狀況，不用每件事都從頭解釋。

快速建立

在 Claude Code 對話框輸入：

/init

Claude 會自動掃描專案，產生一個基本的 CLAUDE.md。

這是最快速的起點，之後可以根據需要自行調整內容

CLAUDE.md 的結構（上）

# 專案名稱

## 專案簡介
一句話描述這個專案是做什麼的

## 技術棧
- 框架：React 18
- 語言：TypeScript
- 樣式：Tailwind CSS
- 狀態管理：Zustand

## 專案結構
src/
├── components/   # UI 元件
├── pages/        # 頁面
├── hooks/        # 自定義 Hooks
├── utils/        # 工具函式
└── types/        # TypeScript 型別

CLAUDE.md 的結構（下）

## 開發指令
- `npm run dev` - 啟動開發伺服器
- `npm run build` - 打包
- `npm run test` - 執行測試

## 程式碼規範
- 使用 ESLint + Prettier
- 元件命名用 PascalCase
- 函式命名用 camelCase
- 檔案命名用 kebab-case

## 注意事項
- 不要直接修改 node_modules
- API 金鑰放在 .env，不要提交到 Git

要放什麼？

內容 為什麼
技術棧 讓 Claude 知道用什麼框架/語言
專案結構 讓 Claude 知道檔案放哪裡
命名規範 讓 Claude 遵守團隊規則
常用指令 方便 Claude 幫你執行
特殊注意事項 避免 Claude 踩雷

不要放什麼？

內容 為什麼
密碼 / API Key 安全風險！會進 Context
超長的文件 浪費 Context 空間
會頻繁變動的內容 維護成本高
其他專案的資訊 會讓 Claude 搞混
絕對不要把機密資訊放進 CLAUDE.md！

CLAUDE.md 的價值

小結

1. 省時間：不用每次都重複說明背景

2. 一致性：Claude 會遵守你的規範

3. 品質提升：有 Context 的回答更精準

4. 團隊共享：新人也能快速用 Claude 協作

2.3 Context 管理實戰

什麼是 Context？

Context = Claude 的「短期記憶」

把 Claude 想像成一個記憶有限的同事：

- 他能記住最近幾小時的對話

- 但記憶空間有上限

- 塞太多東西，舊的會被忘記

查看 Context 使用量

在對話框輸入：

/context

會顯示類似這樣的資訊：

Claude
Context 使用量：45,231 / 200,000 tokens (22.6%)

組成：
- 對話歷史：30,000 tokens
- CLAUDE.md：5,000 tokens
- 引用檔案：10,231 tokens

Context 滿了會怎樣？

Context 使用量：195,000 / 200,000 tokens (97.5%)
問題 影響
早期對話被「遺忘」 Claude 忘記你之前說過什麼
回答品質下降 沒有完整脈絡，答案變不精準
重複問問題 Claude 會問已經討論過的事
複雜任務容易出錯 記不住前面的步驟

策略 1：/compact 壓縮對話

/compact

功能：壓縮對話歷史，保留重點，釋放 Context 空間

什麼時候用：

  • Context 超過 70%
  • 準備開始新任務
  • 之前討論的內容不再需要細節
/compact 會保留對話的「精華」，不會完全清除

策略 2：/clear 清除對話

/clear

功能：完全清除對話，從頭開始

什麼時候用：

  • 完全換一個主題
  • Context 已經很亂
  • 想要「乾淨的開始」
/clear 會完全清除對話！確定不需要之前的對話再用

策略 3：對話分段

不好的做法

一個對話裡做 10 件事

→ Context 爆炸

VS

好的做法

任務 1 → /compact → 任務 2 → /compact → 任務 3

或（不相關時）

任務 1 → /clear → 任務 2

策略 4：精準引用檔案

不好的做法

@src/

把整個資料夾丟進 Context

→ 佔用大量空間

VS

好的做法

@src/components/Button.tsx

只引用需要的檔案

→ 節省空間

Context 工程工具總覽

工具 功能 使用時機
CLAUDE.md 專案長期記憶 專案開始時設定
/context 查看使用量 隨時檢查
/compact 壓縮對話 Context > 70%
/clear 清除對話 換主題、重新開始
@file 精準引用檔案 只加入需要的資訊

2.4 實作練習

練習 1：建立 CLAUDE.md

  • 開啟任意專案資料夾
  • 在 Claude Code 輸入 /init
  • 查看產生的 CLAUDE.md
  • 根據需要修改內容

練習 2：Context 管理

  • 執行 /context 查看目前使用量
  • 進行幾輪對話（問問題、引用檔案）
  • 再次執行 /context，觀察變化
  • 執行 /compact，觀察 Context 變化
  • 執行 /clear，體驗重新開始

核心觀念

1. Prompt vs Context Engineering

Prompt 是單次技巧，Context 是長期基礎

2. CLAUDE.md 是專案的「說明書」

讓 Claude 一開始就知道背景，用 /init 快速建立

3. Context 是有限的

用 /compact 壓縮、/clear 清除，維持效率

4. 精準引用

用 @file 只加入需要的檔案，節省空間
